## 1. 风险评估方法

按 W 模型阶段 1 要求，对识别出的需求风险按「高 / 中 / 低」三级评估，每项风险给出具体缓解措施
与责任阶段。风险评估覆盖技术风险（存储 / 性能 / 安全）、流程风险（覆盖 / 并发）与运营风险
（账号管理 / 部署）。

## 3. 需求冲突检测

### 3.1 冲突对清单

**无冲突。**

5 个 REQ 之间经逐一比对无矛盾：

- REQ-002（认证）与 REQ-003/004/005 无重叠功能，认证为其他模块的前置依赖。
- REQ-003（文章发布）与 REQ-005（审核）在文章状态上有分工：REQ-003 创建时 status=pending，
  REQ-005 审核后变为 approved/rejected；两者状态转移互补，无矛盾。
- REQ-003 文章详情查询（GET /api/articles/:id）与 REQ-005 驳回文章不可见规则一致：
  普通用户查询 rejected 文章返回 404（视为不存在），管理员查询可见全部。
- REQ-004（评论）依赖 REQ-003（文章存在），无循环依赖。

### 3.2 术语一致性检查

| 术语 | 定义 | 出现位置 | 一致性 |
|---|---|---|---|
| 文章状态 | pending / approved / rejected 三态 | REQ-003 §2.1、REQ-005 §2.1、UAT-006/009/010 | ✅ 一致 |
| 管理员 | role=admin 的用户，注册时用户名为 admin 自动获得 | REQ-005 §2.1、NFR-002 §2.2、RISK-004 §2 | ✅ 一致 |
| JWT token | 由 POST /api/auth/login 返回的鉴权令牌，三段式 `xxx.yyy.zzz` | REQ-002 §2.1、NFR-002 §2.2、UAT-004 | ✅ 一致 |
| bcrypt 哈希 | 密码加密方式，cost factor ≥ 10，哈希字符串以 `$2` 开头 | NFR-001 §2.2、RISK-003 §2、UAT-001 | ✅ 一致 |

## 4. 需求缺失项检查

### 4.1 缺失项清单

**无缺失。**

经完整性检查，5 个功能需求 + 6 个非功能需求 + 5 项约束覆盖完整：

| 检查维度 | 状态 | 说明 |
|---|---|---|
| 密码策略 | ✅ | NFR-001 规定 bcrypt 哈希；UAT-003 覆盖空密码边界；未强制复杂度（demo 范围接受，生产环境须补充） |
| 登录失效处理 | ✅ | NFR-002 规定 JWT 过期 ≤ 1 小时；过期/无效 token 返回 401（UAT-007 鉴权失败用例覆盖） |
| 文章状态机 | ✅ | REQ-003/005 明确 pending → approved / pending → rejected 三态；UAT-009/010 覆盖状态转移与可见性 |
| 评论与文章关系 | ✅ | REQ-004 规定评论须关联指定文章；UAT-008 覆盖正常添加；文章不存在时评论返回 404（REQ-004 §2.1 验收标准） |
| 管理员识别 | ✅ | REQ-005 与 RISK-004 明确：用户名 admin 自动获得管理员角色；非管理员调用审核接口返回 403 |

### 4.2 已知限制（已标注，非缺失）

- **无密码复杂度强制**：demo 范围仅校验非空（zod schema），未强制大小写/数字/特殊字符。生产环境须补充。
- **无文章编辑/删除**：REQ-003 仅含发布与查询，无 PUT/DELETE。如需扩展须新增 REQ-006。
- **无评论删除**：REQ-004 仅含添加与查询，无 DELETE。
- **无分页**：文章/评论列表查询未规定分页参数。内存存储数据量小，demo 范围接受；生产环境须补充 `?page=&size=` 参数。
- **无性能基线**：NFR-006 仅声明约束，实测留待阶段 7 系统测试 k6 压测。
