## 1. 项目概述

### 1.1 项目背景

用户需要一个博客系统后端，支持用户发布文章、添加评论，并由管理员审核内容。系统采用
Express 4 + TypeScript 5 (strict) 技术栈，使用内存存储（Map）作为数据持久化，便于端到端
跑通 W 模型 8 阶段开发流程并接受门禁校验。无外部数据库依赖，进程重启数据丢失（已知风险
RISK-001，见 §4）。

### 1.2 项目目标

提供一个最小可用的博客系统后端 API，覆盖：

1. 用户身份认证（注册 / 登录）——基于 bcrypt 哈希与 JWT 鉴权。
2. 文章发布与查询——已登录用户可发布文章，所有人可查询文章列表与详情。
3. 评论添加与查询——已登录用户可对文章添加评论，所有人可查询某文章下的评论列表。
4. 管理员内容审核——管理员可标记文章为通过/驳回，被驳回文章对普通用户不可见。

### 1.3 范围

- 包含：用户注册、用户登录、文章 CRUD（创建/查询）、评论创建/查询、管理员审核接口。
- 不包含：前端 UI、邮件通知、文件上传、富文本编辑器、点赞/收藏、消息推送、性能压测基线
  （留待阶段 7 系统测试）。

## 2. 需求清单

### 2.1 功能需求

| 需求 ID | 模块 | 需求描述 | 优先级 | 验收标准 |
|---|---|---|---|---|
| REQ-001 | 根 | 博客系统根需求（umbrella）：用户可发布文章、添加评论，管理员可审核内容 | 高 | UAT-001~UAT-010 全部通过 |
| REQ-002 | 认证 | 用户身份认证：注册（用户名+密码，bcrypt 哈希存储）与登录（JWT 颁发） | 高 | 注册返回 201 + 用户 ID；登录返回 200 + JWT；密码哈希存储（不可逆）；重复用户名注册返回 409 |
| REQ-003 | 文章 | 文章发布与查询：已登录用户发布文章（标题+正文）；任何人查询文章列表与详情；文章状态含 pending/approved/rejected | 高 | 发布返回 201 + 文章 ID；列表返回 200 + 数组；详情返回 200 或 404；未登录发布返回 401；标题/正文为空返回 400 |
| REQ-004 | 评论 | 评论添加与查询：已登录用户对指定文章添加评论（正文）；任何人查询某文章下评论列表 | 高 | 添加返回 201 + 评论 ID；列表返回 200 + 数组；文章不存在返回 404；未登录添加返回 401；正文为空返回 400 |
| REQ-005 | 审核 | 管理员内容审核：管理员将 pending 文章审核为 approved 或 rejected；rejected 文章对普通用户查询不可见；仅管理员可调用审核接口 | 高 | 管理员审核返回 200 + 新状态；非管理员调用返回 403；普通用户列表查询不返回 rejected 文章；文章不存在返回 404 |

### 2.2 非功能需求

| 需求 ID | 类别 | 描述 | 指标 |
|---|---|---|---|
| NFR-001 | 安全 | 密码须使用 bcrypt 哈希存储，cost factor ≥ 10 | 验证：数据库/Map 中无明文密码，bcrypt 哈希字符串以 `$2` 开头 |
| NFR-002 | 安全 | JWT 鉴权：受保护接口须校验 `Authorization: Bearer <token>` 头；过期/无效 token 返回 401 | JWT 过期时间 ≤ 1 小时；JWT_SECRET 由环境变量注入（测试时 JWT_SECRET=test-secret-blog-demo） |
| NFR-003 | 安全 | 输入校验：所有 POST 接口须使用 zod schema 校验请求体；非法输入返回 400 + 错误明细 | 字段类型/长度校验由 zod 强制执行 |
| NFR-004 | 可用性 | TypeScript strict 模式编译 0 错误 | `tsc --noEmit` 退出码 0 |
| NFR-005 | 可测试性 | 单元测试覆盖率 ≥ 80%（行/分支/函数） | `vitest run --coverage` 报告 lines/branches/functions 均 ≥ 80% |
| NFR-006 | 性能 | 单接口响应时间 < 500ms（内存存储，本地环境） | 阶段 7 系统测试 k6 实测，本阶段仅声明约束 |

### 2.3 约束需求

- **技术栈约束**：后端必须使用 Express 4 + TypeScript 5 (strict)，禁止使用 any/unknown 绕过类型检查；存储必须使用内存 Map，禁止引入 SQLite/PostgreSQL 等外部数据库（便于 demo 自包含）。
- **依赖约束**：仅允许使用 `express` / `bcrypt` / `jsonwebtoken` / `zod` 及对应 `@types/*` 与 `vitest` / `tsx` / `supertest` / `cross-env` / `@vitest/coverage-v8`。禁止引入额外框架（如 NestJS / Koa / ORM）。
- **环境约束**：运行测试须设置 `JWT_SECRET` 环境变量（package.json 已固化 `cross-env JWT_SECRET=test-secret-blog-demo`）。
- **流程约束**：必须遵循 W 模型 8 阶段开发流程；阶段 4 放行前图谱与 TLA+ 须零违反。
- **角色约束**：管理员账号通过注册时预设（如用户名 `admin` 自动获得管理员角色），无独立管理员注册接口。

## 4. 需求风险评估

| 风险 ID | 风险描述 | 等级 | 缓解措施 |
|---|---|---|---|
| RISK-001 | 内存存储重启数据丢失（Map 进程内） | 高 | demo 范围接受；阶段 5 编码时封装存储层接口，便于后续替换为持久化存储；文档中明确告知用户此限制 |
| RISK-002 | JWT_SECRET 泄漏导致 token 伪造 | 高 | 测试环境使用固定 secret（package.json 固化）；生产环境须由部署方注入强随机 secret；禁止将 secret 写入代码或日志 |
| RISK-003 | bcrypt cost factor 过高导致注册/登录延迟 | 中 | cost factor = 10（bcrypt 默认推荐），单次哈希 ~100ms；如阶段 7 性能测试超 500ms 阈值则下调至 8 |
| RISK-004 | 管理员角色判定仅凭用户名 `admin`，易被冒充 | 中 | demo 范围接受；阶段 5 编码时在用户表增加 `role` 字段（'admin'/'user'），注册时若用户名为 admin 则 role=admin；生产环境须改为预创建管理员账号 |
| RISK-005 | 文章审核状态并发修改（管理员同时审核 + 用户查询） | 低 | 内存存储单线程，Express 单实例，无真实并发；阶段 6 集成测试用 supertest 串行调用验证状态转移 |
| RISK-006 | 验收测试用例覆盖不全导致漏测 | 中 | 每个功能需求至少 1 条正常 + 1 条异常 + 1 条边界用例（见 acceptance-test-cases.md）；阶段 8 验收测试执行前 RTM 覆盖率须 100% |
