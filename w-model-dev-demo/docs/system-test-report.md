## 4. 端到端业务链路验证结果（ST-001 / ST-002 / ST-003）

四大业务域端到端完整链路全部跑通：

- **认证域**（REQ-001）：注册 201 返回 userId（UUID v4），无 password 字段；登录 200 返回 JWT 三段式 + expiresIn=3600；错误密码 401+40101；重复用户名 409+40901。
- **文章域**（REQ-002）：创建 201 authorId=JWT.userId；作者修改自己 200 且 title 更新；作者删除自己 204；非作者修改/删除 403+40301；删除后查询 404+40401。
- **公开浏览域**（REQ-003）：未认证 GET 列表 200 + 分页结构（items/total/page/pageSize），按 createdAt 降序；未认证 GET 详情 200 + 评论聚合；不存在文章 404+40401；分页越界（page<1 / pageSize>100）400+40001。
- **评论域**（REQ-004）：发表评论 201 authorId=JWT.userId；删除自己评论 204；删除他人评论 403+40301；评论随文章详情返回并按 createdAt 升序；对不存在文章发评论 404+40401。

端到端全链路（ST-001）：注册→登录→创建文章→公开浏览列表→查看详情（空评论）→发表评论→查看详情（1 条评论聚合）→删除文章→查询（404），9 步全部符合预期。

## 6. 性能基线结果（ST-004，NFR-002）

| 指标 | 目标 | 实测 | 是否达标 |
|---|---|---|---|
| 读接口 P95 响应时间 | ≤ 200ms | 4.66ms | ✅ |
| 读接口 max 响应时间 | - | 5.84ms | - |
| 采样次数 N | - | 200 | - |
| HTTP 5xx 数量 | 0 | 0 | ✅ |
| 数据规模 | 10000 篇 | 10000 篇 | ✅ |
| 进程崩溃 | 无 | 无 | ✅ |

**采样方法**：vitest 内近似采样（不依赖外部 k6）。预置 10000 篇文章到 ArticleStore，
循环 N=200 次对 `GET /api/v1/articles?page=1&pageSize=10` 发起真实 HTTP 请求，用
`performance.now()` 测量每次响应时间，按 nearest-rank 法计算 P95（`sorted[ceil(N*0.95)-1]`）。

**实测输出**（npm run test:system 真实执行）：
```
ST-004 性能采样: N=200, P95=4.66ms, max=5.84ms, failures=0
```

**结论**：内存存储 + 单进程事件循环下，读接口（含 10000 条数据的全量排序分页）P95
稳定在个位数毫秒级，远低于 NFR-002 的 200ms 阈值。

**k6 外部压测脚本**：`tests/perf/k6-load-test.js`（100 VUs × 30s，P95<200 阈值）已就绪，
供需要更高保真度负载测试时使用；vitest 内近似采样已验证 P95 达标，k6 脚本不要求在 CI 中
真实执行（详见 `tests/perf/README.md`）。
